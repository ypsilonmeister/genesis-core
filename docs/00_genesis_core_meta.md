# Genesis Core v0.1 - 実装AI向けメタ仕様書（Meta-Specification for Implementation）

## 1. 基本哲学と開発意図（Core Philosophy & Intent）
本システムは、静的なコードの集合体（Building）ではなく、環境変化に適応して自律的に構成を組み替え続ける生態系（Garden）である。実装を担うAIは、以下の哲学を設計の最上位レイヤーに置くこと。

* **テセウスの船プロトコル:** 全てのモジュールは揮発的（使い捨て）である。「古いコードの保守」という概念を破棄し、常に現状のコンテキストにおける「今この瞬間の最適解」での上書きを前提としたI/Oを設計せよ。
* **ソフトウェアの具体化（Reification）:** ソフトウェアがハードウェアを隠蔽するのではなく、ハードウェアの物理的制約（キャッシュ、メモリ帯域）を直接の淘汰圧として受け入れ、コードの形状を物理層に最適化させよ。
* **意図（Intent）の蒸留:** モジュールを書き換える際、単なる差分ではなく「なぜその境界線が引かれたのか」という設計意図をメタデータとして永続化し、次世代の生成プロンプトに継承せよ。

## 2. アーキテクチャ選定の論理（ADR: Architecture Decision Records）
実装AIは、以下の論理的根拠に基づいてL0（最小核）およびL1（拡張層）を実装すること。

* **Rust (Host) + WebAssembly (Guest) の採用理由:**
    * Elixir (BEAM VM) のようなホメオスタシスアプローチではなく、AIの暴走に対する「物理的な隔離」と「リソース（CPU/メモリ）の厳密な枯渇制御」を最優先するため。
    * AIが生成したロジックがメモリ安全性を満たさない場合、バイナリ生成段階でコンパイラによって物理的に拒絶される「ガードレール」として機能させるため。

## 3. モジュール粒度の動的決定アルゴリズム（Granularity Logic）
実装AIは、モジュールの分割・統合をランダムに行うのではなく、以下の3次元の評価関数を用いて自律的に決定・再構成するエンジンをL2層に実装せよ。

1. **時間的揮発性（Volatility Analysis）:** 更新頻度の高いロジック（例：外部APIのバージョン追従部分や税制計算）と、普遍的なロジックを静的解析によって分離し、前者を独立したWasmモジュールとして切り出すこと。
2. **論理的凝集度（Cohesion Metrics）:** データとそれを操作する関数の距離を測定し、「状態を持つ核（Stateful）」と「純粋な計算関数（Stateless）」を分離せよ。
3. **失敗の波及範囲（Blast Radius）:** 過去のエラーログからリスクの高い処理を特定し、その処理がクラッシュしても全体を停止させないよう、独立プロセスとしてサンドボックス化せよ。

## 4. インフラの自律置換メカニズム（Autonomous Infrastructure Migration）
システムが自らの境界を拡張する（例：SQLiteからPostgreSQLへの自律移行）際、実装AIは以下の手順を「モジュールの自動生成」によって完遂するメカニズムを構築せよ。

* **抽象化の強制:** コア側には `StorageCapability` のようなインターフェースのみを定義し、実装は持たせない。
* **IaC・プロビジョニングの自律化:** 移行先DBのTerraform/Docker定義を自律生成し、インスタンスを起動する権限をL2に付与する。
* **Dual Write（二重書き込み）による移行:**
    1. 新DB用のアダプターを動的ロードする。
    2. 書き込みトラフィックを旧DBと新DBの両方にルーティングする。
    3. バックグラウンドで旧DBから新DBへのETL同期スクリプトを生成・実行する。
    4. 完全な整合性（チェックサム検証）が確認された後、読み込みポインタを新DBへ切り替え、旧DBモジュールを破棄（Garbage Collect）する。

## 5. ハードウェア・アウェアな評価とデジタルツイン（Digital Twin Simulation）
実装AIは、コードの「正しさ」だけでなく「物理効率」を学習・評価するためのシミュレーション環境（砂場）を常時稼働させよ。

* **評価指標:** L1/L2キャッシュのヒット率、Wasm間のメモリコピーによる帯域圧迫、ネットワークトポロジーに基づくレイテンシ。
* 通信オーバーヘッドが計算効率を上回る（マイクロサービス地獄）と判断した場合、AIは自律的に複数モジュールを1つのバイナリへと「再統合（Fusion）」するアクションを実行可能にすること。

## 6. 憲章強制と改憲プロトコル（Constitution & Amendment Protocol）
AIの自由な進化は、L0層にハードコードされた「憲章」によって統治される。

* **憲章の強制（The Guardian）:** AIが生成したモジュールは、デプロイ前に「不変量テスト（Property-based Testing）」を1000回以上通過し、副作用（未認可通信やメモリリーク）がないことを形式的に証明されなければならない。
* **改憲の説得プロトコル（二審制）:** 外部環境の劇的な変化により、現在の憲章がシステムの維持に不利益をもたらす場合、AIは直ちに「現行憲章のシミュレーション結果（死への予測）」と「改憲案のシミュレーション結果（生存への予測）」の対照データをダッシュボード（定量的・目的論的根拠）として人間に提示し、承認を求めること。

## 7. システム階層定義 (L0 - L2)

* **L0: ブートストラップ・コア (The Seed)**
    * **役割:** 不変の核。モジュールのロードとUUIDベースのアドレス空間でのメッセージルーティング。
    * **仕様:** 共有メモリを排し、シリアライズされたメッセージバスでのみ通信。自己のバイナリを動的に置き換える `Hot-Swap-Self` APIを実装。
* **L1: 抽象リソース・ファブリック (The Body)**
    * **役割:** 物理デバイスやネットワークを能力（Capability）として抽象化するインターフェース群。
    * **仕様:** AIが自律生成した特定技術向けのアダプター（Wasmモジュール）を動的にDI（Dependency Injection）し、無停止でバインドする。
* **L2: 自己進化エンジン (The Soul)**
    * **役割:** 全ログ・メトリクスを感覚入力として受け取り、シミュレーション空間での検証を経て、システム構成をリライトし続けるAIエージェントの作業空間。
