# Cognitive Module Protocol (CMP) v0.2
## — AI認知単位を基準とするソフトウェア・アーキテクチャ仕様書

---

## 0. 一行要約

**ソフトウェアのモジュール境界を「人間の認知」ではなく「LLMの認知」を基準に切り直し、創成期は人間+AIで書き、保守期はAIがRustコンパイラとシステム憲章の二重ガードレールの下で自律改変するシステムを構築する。**

---

## 1. 動機(Why)

### 1.1 コンウェイの法則のAI版

> 「ソフトウェアのアーキテクチャは、それを書き換える主体の認知構造を反映する」

従来のモジュール原理——構造化プログラミング、オブジェクト指向、マイクロサービス、DDD——はすべて**人間の認知**を基準にしてきた。可読性、変更容易性、組織構造との整合性。

AIがコードを書き換える時代には、**AIの認知**を基準にした単位が必要である。CMPはこの仮説に基づくアーキテクチャ提案である。

### 1.2 解こうとしている問題

- AI生成コードの世代継承で意図が漂流する問題
- AI自律改変が単一メトリクスに過適合する問題
- モジュール粒度の決定が事後正当化に陥る問題
- AI生成コードのメモリ安全性・データ競合・リソース暴走の物理的防止

### 1.3 解かないことに決めた問題

- 完全な無人運用(Tier 3は人間承認を残す)
- 大規模分散システムへの適用(まずは単一ホスト内)
- ハードウェア物理層への最適化(キャッシュ・帯域最適化はPhase 3以降)

## 2. 中核概念

### 2.1 AI認知モジュール (Cognitive Module)

「LLMが意味的に把握しきれる単位」で切られたモジュール。判定基準は2つ:

**物理制約**: モジュール本体 + 個別憲章 + 隣接モジュールのインターフェース + 直近の変更履歴が、LLMの実用的コンテキスト(20K token程度)に収まること。

**意味制約**: LLMがそのモジュールを1〜2文で要約でき、かつ要約が個別憲章のWhatと意味的に一致すること。

### 2.2 個別憲章 (Module Charter)

各モジュールのヘッダに自然言語で書き込まれる5項目:

| 項目 | 内容 |
|---|---|
| **What** | このモジュールは何をするか(1文) |
| **Invariants** | 絶対に破ってはいけない性質 |
| **Boundaries** | 依存先と被依存先 |
| **Extensible** | 変更を許す範囲 |
| **Why** | このモジュールが存在する理由 |

個別憲章はAIの改変対象だが、改変には全体憲章との整合性チェックが必要(§4.2)。

### 2.3 システム憲章 (System Charter)

システム全体を統治する独立ファイル。**保守期にAIが書き換えてはならない**(改変は人間承認のみ)。

#### 2.3.1 システム憲章の9項目

| 項目 | 内容 |
|---|---|
| **1. Purpose** | このシステムは何のために存在するか(1段落) |
| **2. Hard Invariants** | どのモジュールも破ってはいけない性質(データ損失禁止、特定API呼び出し禁止、リソース上限など)。違反は実行時に物理的にブロックされる |
| **3. Soft Invariants** | 推奨される不変条件。違反する場合は記録と人間承認が必要 |
| **4. Module Addition Criteria** | Tier 2でAIが新モジュールを作ってよい条件 |
| **5. Module Deletion Criteria** | Tier 3で削除を提案してよい条件 |
| **6. Tier Boundaries** | 各Tierの権限境界。曖昧ケースの分類規則 |
| **7. Failure Mode Declarations** | 各種失敗パターンへのフォールバック方針 |
| **8. External Dependencies** | 外部サービス、APIキー、データソース。これらの変更は人間承認必須 |
| **9. Amendment Process** | システム憲章自体を改訂する手順 |

#### 2.3.2 二層構造: 読み物としての憲章 + 執行装置としての憲章

システム憲章は二層構造を持つ。

**Layer A: 自然言語層(Reading Layer)**

人間とAIが読むためのMarkdownドキュメント。AIは改変提案時にこれを参照し、自分の提案が憲章に違反していないかを自己チェックする。

**Layer B: 実行可能規則層(Enforcement Layer)**

Hard InvariantsとExternal Dependenciesの一部は、**実行時に機械的に強制される検証規則**としてRustコードで実装される。

例:
- 「データ損失禁止」 → 全てのWrite操作はトランザクション境界内でのみ許可、Commitなしの破棄は実行時例外
- 「外部API呼び出し制限」 → 許可ドメインリストをオーケストレータが検閲、未許可通信は物理的にブロック
- 「メモリ上限」 → 各モジュールプロセスに ulimit / cgroups で物理制限

**Layer AとLayer Bの一貫性**は、システム憲章改訂時に人間が責任を持つ。Layer Aを変更したらLayer Bも更新する。この一貫性はAIの責務範囲外。

## 3. 二相モデル

CMPは時間軸で2つの相を持つ。

### 3.1 創成期 (Genesis Phase)

人間とAIが協働してシステムを構築する期間。

- 人間が要件と**システム憲章の初版**を起草(特にHard Invariantsを慎重に定義)
- Layer B執行コードも創成期に書く
- AIが対話的にモジュールを生成
- 各モジュールには個別憲章を必ず埋め込む
- 人間が「v1としてリリース」を宣言した瞬間、保守期へ移行

### 3.2 保守期 (Stewardship Phase)

AIが自律的に改変を行う期間。改変は§4の三段階権限に従う。

人間は以下に関与:
- Tier 3変更の承認
- システム憲章の改訂(Layer A + Layer B両方)
- 過適合警報への応答

---

## 4. 三段階権限モデル

AI自律改変は3段階に区切られる。下位ほど自動化される。

### 4.1 Tier 1: モジュール内改変 (自動)

**対象**: バグ修正、リファクタ、パフォーマンス改善

**条件**:
- 個別憲章のInvariantsを破らない
- 個別憲章のBoundariesを変更しない
- システム憲章のHard Invariantsを破らない
- 既存テストが全て通る
- 新たに自動生成された性質テストも通る
- **Rustコンパイラを通る**(物理的検閲)

**プロトコル**:
1. エラーログまたはメトリクスが改変トリガを発生
2. AIが個別憲章+モジュール本体+トリガ情報を入力に改変案を生成
3. Rustとしてビルド(失敗なら即却下)
4. サンドボックスでテスト実行
5. Layer B実行可能規則による検閲
6. 全て通過すれば hot swap(プロセス入替、§8参照)
7. 全工程をメタデータとして§6.1に転写

### 4.2 Tier 2: モジュール追加 (自動)

**対象**: 新機能要求、新外部システム対応

**条件**:
- システム憲章のModule Addition Criteriaを満たす
- 既存モジュールのInvariantsに影響しない
- 新モジュールの個別憲章をAIが自動生成
- システム全体テストが通る

**プロトコル**:
1. 機能要求の入力(自然言語または定型API)
2. AIが既存モジュール群への影響分析
3. 新モジュール本体+個別憲章を生成
4. **対照群: 既存モジュール拡張案も並行生成**(モジュール追加が本当に必要かの検証)
5. 両案をサンドボックスで評価、優位な方を採用
6. 採用案がデプロイ、却下案も§6.1に転写

### 4.3 Tier 3: モジュール境界再編 (人間承認必須)

**対象**: モジュールの削除、統合、分割、境界の再定義

**理由**: 境界変更は認知単位そのものの変更であり、影響範囲が読みきれない。

**プロトコル**:
1. AIが境界再編案を生成
2. **四案比較**を人間に提示:
   - A: 現状維持
   - B: AI提案
   - C: ランダム再編案(プラセボ対照)
   - D: 逆方向の再編
3. 各案のシミュレーション結果を並べる
4. BがAおよびCに対して明確な優位を示せない場合、Guardian がBを却下
5. 人間が承認したもののみ実行

## 5. モジュール境界の自己検証

CMPの中核仮説——「AI認知単位でモジュールを切る」——は、それ自体が検証可能である。

### 5.1 把握テスト (Comprehension Test)

定期的に各モジュールに対して実行:

1. 別インスタンスのLLMにモジュール本体のみを渡す(個別憲章は隠す)
2. 「このモジュールを1文で要約せよ」と問う
3. 要約と個別憲章のWhatを、第三のLLMが意味的一致判定
4. **不一致が続くモジュールは分割候補**

### 5.2 統合テスト (Cohesion Test)

隣接する2モジュールに対して:

1. 両方をLLMに見せて「1モジュールにすべきか」を問う
2. 複数モデルで多数決
3. 統合推奨が連続して出るペアは、Tier 3で人間に統合案を提示

### 5.3 自己検証の対照群

判定LLM自体が漂流していないか確認するため、**人間が定期的に少数サンプルをレビュー**(週1回、5モジュール程度)。LLM判定と人間判定の乖離が拡大したら、判定基準を再調整する。

---

## 6. 意図メタデータ

### 6.1 転写原則 (Transcribe, Not Edit)

すべての改変は以下を**無編集で**メタデータストアに追記する:

- 改変時刻
- トリガ(エラーログ、要求、メトリクス)
- 生成プロンプト全文
- 使用AIモデル(名称+バージョン)
- 生成された改変案全文
- ビルド結果、テスト結果、Layer B検閲結果
- 採否結果と理由
- (Tier 3の場合)四案比較結果

このメタデータは**追記専用**であり、編集・要約・削除を行わない。

### 6.2 次世代AIへの継承

次回の改変提案時、AIに渡されるのは:
- 現在のモジュール本体+個別憲章
- システム憲章(Layer A)
- **直近N回分の改変メタデータ(無編集の生データ)**

AIがメタデータを「整理してから後続に渡す」ことは禁ずる。

---

## 7. 過適合への防御

### 7.1 メトリクス過適合への対処

- 改変提案時に「最適化する目的関数」を**事前宣言**させる
- 同一モジュールに対する連続改変で目的関数が変動したら警報
- Tier 2は対照群との比較を強制
- Tier 3は四案比較を強制

### 7.2 判定LLMへの過適合への対処

- 把握テスト・統合テストは複数モデルで実施
- 判定モデルの結果と人間の少数レビューを定期突合(§5.3)

### 7.3 自己改変への過適合への対処

以下は**不可侵領域**であり、AIの改変対象外:

- システム憲章 Layer A(自然言語ドキュメント)
- システム憲章 Layer B(実行可能検証規則のRustコード)
- §5の自己検証ロジック
- §6のメタデータ追記ロジック
- §8.3のオーケストレータ本体

これらがシステムの錨となる。

## 8. 実装アーキテクチャ (Rust採用)

### 8.1 Rust採用の根拠 (ADR-CMP-001)

**採用理由**:
- **AI生成コードの物理的検閲**: メモリ安全性違反、データ競合、未定義動作をコンパイル時に拒絶。AIの暴走に対する最初のガードレール。
- **リソース制御の精度**: cgroups/ulimit/Wasmランタイムを通じた厳密な資源制限。
- **Hot swap可能性**: プロセス分離方式およびWasm方式の双方が実現可能。

**放棄した選択肢と理由**:
- **Elixir/OTP**: Hot swapはネイティブだが、メモリ安全性は型システム任せ、AI生成コードの暴走を物理的に止めにくい。
- **Python**: 実装速度は速いが、実行時まで多くの問題が顕在化しない。ガードレールが弱い。
- **Go**: 中庸だが、Rustほどのコンパイル時保証はなく、Wasmサンドボックスのエコシステムが弱い。
- **C/C++**: 性能はあるが、AI生成コードを通すには危険すぎる。

**前提が崩れた場合の再評価条件**: AIコード生成の品質が劇的に向上し、メモリ安全性違反が実質的にゼロになった場合、Pythonへの移行を検討。

### 8.2 二段階アーキテクチャ

**Phase 1: Pure Rust + プロセス分離(リーンスタート用)**

- 各モジュールは独立Rustバイナリ
- 親プロセス(オーケストレータ)がモジュール起動・通信・差替を管理
- 通信は Unix Domain Socket + MessagePack
- Hot swap手順:
  1. 新バイナリをビルド(コンパイル失敗時点で却下)
  2. テスト実行
  3. 新プロセス起動
  4. ヘルスチェック通過後、トラフィック切替
  5. 旧プロセス graceful shutdown
- Layer B検閲はオーケストレータがプロセス間通信時に実施

**Phase 2: Rust Host + Wasm Guest(発展形)**

- Phase 1の通信プロトコルを維持
- モジュール本体を Wasm(wasmtime ランタイム)に置き換え可能
- 利点:
  - リソース制限の精密化(fuel-based metering)
  - メモリ分離が物理的に強い
  - マルチ言語化(将来Python/AssemblyScript等のWasmコンパイル対応)
- Phase 1で動いたモジュールは段階的に Wasm 化可能

**Phase 3: ハードウェア最適化(将来)**

SPEC v0.1にあったキャッシュヒット率・メモリ帯域最適化はここで検討。Phase 2が安定運用に到達してから着手。

### 8.3 オーケストレータの責務

オーケストレータは**AI改変対象外**の不可侵領域。以下を担う:

- モジュールプロセスのライフサイクル管理
- モジュール間通信のルーティング
- Layer B(実行可能検証規則)の執行
- 改変提案AIの呼び出し
- メタデータストアへの書き込み
- 自己検証ループ(§5)の実行
- Tier 3提案時の人間への通知

### 8.4 推奨ライブラリ構成

- **オーケストレータ**: `tokio` (非同期ランタイム), `serde` + `rmp-serde` (MessagePack)
- **モジュール間IPC**: `tokio::net::UnixListener` / `UnixStream`
- **メタデータストア**: `rusqlite` (SQLite)
- **テスト実行**: `cargo test` をプロセス実行で呼び出し
- **AI API**: `reqwest` で Anthropic / Gemini API
- **ビルドサンドボックス**: 一時ディレクトリ内で `cargo build` を分離プロセス実行

### 8.5 リーンスタートのロードマップ

**Week 1: 創成期ワークフローと骨格**
- 題材選定(推奨: RSSアグリゲータ)
- システム憲章9項目の起草、Layer B最小実装
- 個別憲章ヘッダフォーマット確定
- オーケストレータ骨格、最初の手動モジュール

**Week 2: Tier 1ループ実装**
- パーサー破損の検出と自動修復
- Rustビルドサンドボックス
- メタデータ転写

**Week 3: Tier 2ループ実装**
- 新サイト対応モジュールの自動生成
- 対照群評価
- システム憲章のAddition Criteriaチェック

**Week 4: 自己検証と観察**
- §5の把握テスト・統合テスト実装
- 30日連続運転、漂流の有無を観察

### 8.6 評価指標

- Tier 1自動修復の成功率
- Tier 2提案と対照群の優劣分布
- 把握テストの一致率の時系列推移
- システム憲章違反の発生回数(Hard/Soft別)
- Layer B検閲のブロック回数
- Rustコンパイラ却下回数(AIの「物理的暴走」頻度)
- 人間介入が必要になった頻度

## 9. 失敗の予期

CMPが失敗するとしたら、最も可能性が高いシナリオ:

1. **把握テストが機能しない**: LLMが要約能力と判定能力を持たない、または評価者効果が大きすぎる
2. **個別憲章が形骸化する**: AIが憲章を儀礼的に書き、実態と乖離していく
3. **Tier境界が曖昧化する**: 「Invariant違反か拡張か」の判定が揺れる
4. **メタデータが肥大化して活用不能**: 数万行のログでAIが参照不能になる
5. **Rustビルド失敗が多発し改変速度が崩壊**: AI生成Rustコードのコンパイル成功率が低すぎる
6. **Layer A と Layer B の乖離**: 人間が憲章改訂時に同期を怠り、二層が食い違う

対処は実物を見てから判断する。**先回りして仕様に書き込まない**。

## 10. 改訂履歴

- **v0.1 Genesis Core**: 「自律進化するシステム」の壮大な構想。Rust+Wasm、デジタルツイン、L0/L1/L2三層、Dual Write、四案比較改憲プロトコル等を含む。
- **v0.2 Genesis Core**: KSAU過適合経験を反映し、対照群と過適合監視を追加。過適合監視自体が過剰実装になる矛盾を内包。
- **v0.1 CMP**: 「核は何か」を問い直し、モジュール化・二相モデル・三段階権限を中心に再構成。Wasm・デジタルツイン・自律インフラ移行を削除。Tier 3の四案比較のみ保持。実装スタックはElixir/Pythonを想定。
- **v0.2 CMP (本仕様)**: システム憲章を9項目フォーマットで具体化、二層構造(Layer A/B)を導入。実装スタックをRustに変更し、Phase 1(Pure Rust + プロセス分離) → Phase 2(Rust + Wasm) → Phase 3(ハードウェア最適化)の段階構成を明示。

## 付録A: CMPと既存技術の関係

| 既存技術 | CMPとの関係 |
|---|---|
| Erlang/OTP hot swap | 思想的先祖。CMPはRustプロセス分離で類似機能を実現 |
| LISPイメージベース開発 | 思想的先祖だが、書き換え主体が人間 |
| マイクロサービス | モジュール独立性は近いが、CMPはAI認知単位で切る |
| DDD | ドメイン境界は人間認知基準、CMPはAI認知基準 |
| Property-based Testing | Tier 1の自動検証に組み込む |
| LLM-as-judge | §5の自己検証で使用、判定モデル多重化で漂流対策 |
| WebAssembly + Wasmtime | Phase 2で採用、AI生成コードの完全サンドボックス化 |
| Rust ownership system | Tier 1の物理的検閲機構 |

## 付録B: 用語

- **Cognitive Module**: AI認知単位で切られたモジュール
- **Module Charter**: 個別憲章(モジュールヘッダの5項目)
- **System Charter**: システム憲章(9項目、Layer A + Layer B)
- **Layer A (Reading Layer)**: 自然言語のシステム憲章ドキュメント
- **Layer B (Enforcement Layer)**: 実行可能検証規則
- **Genesis Phase**: 創成期
- **Stewardship Phase**: 保守期
- **Tier 1/2/3**: AI自律権限の三段階
- **Comprehension Test**: 把握テスト
- **Cohesion Test**: 統合テスト
- **Transcribe Not Edit**: メタデータの転写原則
- **Orchestrator**: 不可侵領域のシステム制御プロセス
